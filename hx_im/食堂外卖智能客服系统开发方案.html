一、项目概述与背景

1.1 业务场景

食堂外卖智能客服系统服务于外卖平台的终端用户，核心场景包括：

  • 售前咨询：菜品口味、价格、商家信息、优惠活动
  • 订单查询：配送状态、订单详情、历史订单
  • 售后处理：取消订单、退款申请、投诉建议
  • 问题解决：错发漏发、食品质量、转人工升级

1.2 当前系统定位

本方案基于浣熊食堂智能客服系统现有架构进行诊断与演进设计。现有系统已具备：

  • 13 步串行 Pipeline 编排
  • 20+ 意图识别（两级：向量相似度 + LLM CoT 推理）
  • 8 个业务函数（查询 3 + 操作 5）
  • ChromaDB 三层记忆架构（短期/工作/长期）
  • Agent Loop 多步自主执行（可选开启）
  • 意图路由 RAG（products / faq / policies 三集合）
  • 声明式 JSON 配置（intents.json / functions.json / sop.json）

1.3 核心痛点

问题 影响
13 步串行 Pipeline，意图识别与槽位提取互不依赖却串行执行 P50 延迟 2-5s
Step 0-5 全链路串行，Router 决策后才能进入知识/函数执行 高并发下延迟恶化明显
Agent Loop 仅在 Step 8.5/9 内循环，能力单一 复杂多步场景效果受限
单一 LLM 调用链路长 Token 成本高，难以独立优化各环节
转人工判断依赖规则（5 条件） 灵活性和召回率受限

二、主流范式深度对比

2.1 范式一览

范式 核心思路 P50 延迟 成本 完成率 复杂度
串行 Pipeline 13 步线性执行，单一 LLM 2-5s 中 60-75% 低
Multi-Agent 消息总线 各 Agent 异步通信，共享消息队列 3-8s 高 75-90% 高
Hierarchical Coordinator Router 分类 → 分发给专业 Agent 2-5s 中高 70-85% 中高
LLM-as-Judge 路由 按复杂度动态分流到不同处理链路 +0.3s 低 +5-15% 中
Hybrid Coordinator（推荐） Router-first + 动态路由 + 共享记忆 1.5-4s 中 80-92% 中高

2.2 各范式详细对比

范式 A：串行 Pipeline（当前浣熊系统）

用户消息 → Step0 → Step0.5 → Step1 → Step1.5 → Step2 → Step3 → Step4 → Step5 → Step6 → Step6.5 → Step7 → Step8 → Step8.5 → Step9 → Step10 → 返回用户
  • 优点：流程清晰、便于监控、排查问题容易、新增意图只需改 JSON 配置
  • 缺点：无法并行（意图识别和槽位提取互不依赖）、单一 LLM 链路耦合重、无法独立优化各环节
  • 适用：意图收敛（<50 种）、单轮或多轮简单流转、团队初期快速验证

范式 B：Multi-Agent 消息总线

消息队列（Redis/RabbitMQ）
  ├── Intent Agent ──→ 意图分类结果
  ├── Slot Agent  ──→ 槽位提取结果
  ├── RAG Agent   ──→ 知识检索结果
  ├── Action Agent ──→ 函数执行结果
  └── Memory Agent ──→ 记忆更新结果
                    ↓
              Generator Agent
                    ↓
                 返回用户
  • 优点：容错好（一个 Agent 挂了其他继续）、可独立扩缩容、并行执行理论上能到 1-2s
  • 缺点：消息顺序保障复杂、Agent 间状态同步困难、调试成本极高、需要成熟中间件团队
  • 适用：高并发、复杂业务、有中间件运维能力

范式 C：Hierarchical Coordinator

Router Agent（接管原 Step 0-5）
  ├── 意图分类（快）
  ├── 阶段判断
  ├── 情感检测
  └── 转人工判断
        │
        ├── KNOWLEDGE → 知识检索 Agent → Generator
        ├── ACTION    → 函数执行 Agent → Generator
        └── HYBRID    → 知识 + 函数并行 → Generator
  • 优点：清晰分层结构、便于监控每个 Agent、逐个优化、独立扩展
  • 缺点：Router 是核心瓶颈、必须保证高准确率
  • 适用：生产环境首选

范式 D：LLM-as-Judge 动态路由

用户消息
    │
    ▼
Router LLM（轻量，<100 tokens）
    │
    ├── 简单   → 知识库直接检索 → 返回（<500ms）
    ├── 中等   → 意图识别 → 槽位 → 返回（1-2s）
    └── 复杂   → 走完整 Pipeline / Multi-Agent（3-5s）
  • 优点：成本效率极高（节省 40-60% Token）、用户体验分层
  • 缺点：Router 分类准确率是整个系统的瓶颈（5-15% 误分类率）
  • 适用：成本敏感、有明确”简单/复杂”区分度的场景

范式 E：Hybrid Coordinator（推荐落地方案）

整合以上范式优点，按场景自适应：

用户消息
    │
    ▼
┌─────────────────────────────┐
│  Router Agent（~300ms）      │  ← 接管原 Step 0-5，合并为一次 LLM 调用
│  意图识别 + 阶段判断 +      │
│  情感检测 + 转人工判断       │
└──────────┬────────────────┘
           │
   ┌───────┼───────┬──────────┐
   ▼       ▼       ▼          ▼
KNOWLEDGE  ACTION  HYBRID   CHITCHAT
 (~500ms) (~1-2s) (~2-3s)  直接回复
   │       │       │          │
   └───────┼───────┘
           ▼
    Generator Agent + Memory Agent（共享三层记忆）
           │
           ▼
        返回用户

核心优势： 1. 并行执行 — KNOWLEDGE / ACTION / HYBRID 可并行检索知识+执行函数 2. 职责单一 — 每个 Agent 只专注一件事，LLM 调用更精准 3. 独立扩展 — 哪个环节慢就单独优化，不影响全链路 4. 统一记忆 — Memory Agent 作为共享记忆中枢，避免各 Agent 上下文割裂 5. 容错隔离 — 某个 Agent 失败不影响其他，Pipeline 改为消息总线兜底

3.2 Agent 职责定义

Agent 接管原 Pipeline 输入 输出 核心能力
Router Agent Step 0-5 用户消息 + session_id 意图 + 槽位 + 阶段 + 情感 + 路由决策 一次 LLM 调用并行输出多维信号
Knowledge Agent Step 6 + 8 意图 + 槽位 RAG 检索结果（top-3） 意图路由选择集合（products/faq/policies）
Action Agent Step 6 + 8.5 意图 + 槽位 函数执行结果 8 业务函数 + Agent Loop 多步执行
Hybrid Agent Step 6 + 8 + 8.5 意图 + 槽位 知识片段 + 函数结果 知识检索与函数执行并行协作
Generator Agent Step 9 + 10 所有上游结果 最终回复 SOP 指令 + 记忆上下文 + RAG + 函数 → LLM 生成
Memory Agent Step 0 + 10 session_id + 关键事实 记忆召回 / 压缩 / 持久化 三层记忆（短期/工作/长期）+ ChromaDB

3.3 路由决策逻辑

class RouterDecision:
    def decide(self, intent, stage, emotion, slots, confidence) -> str:
        # 1. 转人工优先
        if self._should_escalate(emotion, confidence, stage):
            return "ESCALATE"

        # 2. 闲聊直接回复
        if intent in ["chitchat", "greeting", "goodbye"]:
            return "CHITCHAT"

        # 3. 知识查询为主
        if intent in ["product_inquiry", "price_inquiry", "faq", "policy_inquiry"]:
            if len(slots) < 2:
                return "KNOWLEDGE"  # 简单知识查询
            else:
                return "HYBRID"     # 知识 + 少量槽位

        # 4. 操作/执行类
        if intent in ["order_cancel", "refund_request", "modify_order", "complaint"]:
            return "HYBRID"  # 需要函数执行，但可能也需要知识支撑

        # 5. 纯操作
        if intent in ["delivery_inquiry", "rush_order"]:
            if len(slots) >= 1:
                return "ACTION"   # 槽位足够，直接执行
            else:
                return "HYBRID"   # 槽位不足，需要追问

        # 6. 默认
        return "HYBRID"

四、演进路线图

Phase 1：Router-first 改造（2-4 周）

目标：最小改动，提升 Router 决策效率

改动： - 将 Step 0-5 合并为 Router Agent，一次 LLM 调用并行输出意图 + 阶段 + 情感 + 转人工 - Step 6-10 保持不变

预期效果： - P50 延迟：2.5s → 1.8s（节省 ~700ms） - 意图准确率：+3-6%（Router 一次 LLM 调用，减少信息损失） - Token 消耗：减少约 20%（减少了串行 LLM 调用次数）

Phase 2：动态路由（4-8 周）

目标：按路由并行执行 KNOWLEDGE / ACTION

改动： - Router 输出路由决策（KNOWLEDGE / ACTION / HYBRID / CHITCHAT） - 按路由分发到对应 Agent，并行执行 - Generator Agent 汇总结果

预期效果： - P50 延迟：1.8s → 1.2s（知识检索 + 函数执行并行） - 任务完成率：68% → 75-78%（路由分流，更精准的处理链路） - 转人工召回率：70% → 80-85%（Router LLM 判断 > 规则判断）

Phase 3：Hybrid Agent + Memory 共享（8-16 周）

目标：完整 Multi-Agent 协作

改动： - Hybrid Agent：知识检索与函数执行并行协作 - Memory Agent：统一管理三层记忆，作为共享中枢 - Agent Loop 升级：在各 Agent 内支持多步自主执行 - 可观测性：OpenTelemetry 全链路追踪

预期效果： - 任务完成率：75-78% → 80-85% - P99 延迟：改善（容错隔离，单 Agent 失败不影响整体） - 可独立优化各 Agent，不影响全链路

Phase 4：持续优化（长期）

  • Benchmark 持续评测，定位瓶颈
  • 按评测结果优化各 Agent Prompt
  • 引入 Human-in-the-loop 反馈闭环
  • 长尾意图专项优化

六、生产落地关键要素

6.1 可观测性（生死线）

生产环境必须建设全链路追踪：

# OpenTelemetry 埋点示例
from opentelemetry import trace

 tracer = trace.get_tracer(__name__)

async def chat_flow(query: str):
    with tracer.start_as_current_span("router_agent") as span:
        span.set_attribute("query", query)
        intent = await router.decide(query)
        span.set_attribute("intent", intent)

    with tracer.start_as_current_span(f"{intent}_agent"):
        result = await execute_agent(intent, query)

    with tracer.start_as_current_span("generator_agent"):
        reply = await generator.generate(result)

关键 Trace 链路：trace_id → router → knowledge/action/hybrid → generator → memory

推荐工具：OpenTelemetry + LangSmith / Arize / Helicone

6.2 成本控制

策略 说明 预期节省
意图分类前置 小模型（GPT-4o-mini / DeBERTa）做 Tier-1 分类 40-60% Token
上下文截断 超过 128K 后主动压缩对话历史 避免成本爆炸
路由分流 简单问题不走完整 Pipeline 50-70% 简单请求节省
缓存复用 相同意图+槽位的结果缓存 减少重复 LLM 调用

6.3 安全与合规

风险 应对措施
Prompt 注入 输入清洗 + 长度限制
隐私泄露 PII 检测（手机号/地址等）脱敏
过度承诺 回复生成增加安全过滤层
模型幻觉 RAG 结果必须带来源标注

6.4 人工兜底

再好的 Agent 也有 8-15% 无法解决。必须内置：

  • 主动识别：置信度 < 0.6 + 连续负面情绪 + 超出范围 → 立即转人工
  • 被动兜底：用户明确要求转人工
  • 反馈闭环：人工处理结果回流到训练数据

class CustomerServiceState(TypedDict):
    """客服系统共享状态"""
    # 基础信息
    query: str                          # 用户原始消息
    session_id: str                     # 会话 ID
    user_id: str                        # 用户 ID

    intent: str                         # 意图分类结果
    stage: str                          # 阶段（售前/售中/售后）
    emotion: str                        # 情感（正面/中性/负面）
    confidence: float                   # 置信度
    escalate: bool                      # 是否转人工

    # 路由决策
    route: str                          # KNOWLEDGE / ACTION / HYBRID / CHITCHAT / ESCALATE

    # 下游 Agent 结果
    slots: dict                        # 槽位提取结果
    rag_results: list                   # RAG 检索结果
    function_results: dict              # 函数执行结果

    # 最终输出
    reply: str                          # 最终回复
    memory_updates: dict                # 记忆更新

    # 追踪信息
    latency_ms: float
    tokens: int

接管原 Pipeline Step 0-5：记忆召回、文本纠正、意图识别、话题切换、阶段判断、情感检测、转人工判断。

router_chain = router_prompt | llm.with_structured_output(RouterOutput)

@tool
def router_agent(state: CustomerServiceState) -> CustomerServiceState:
    """Router Agent：合并 Step 0-5，一次 LLM 调用输出多维信号"""
    # 1. 召回三层记忆
    memory = recall_memory(state["session_id"], state["user_id"])

    # 2. Router LLM 调用
    result = router_chain.invoke({
        "query": state["query"],
        "history_summary": memory["summary"],
        "user_profile": memory["profile"],
        "intent_list": load_intents()  # 从 intents.json 加载
    })

    # 3. 路由决策
    route = decide_route(result["intent"], result["stage"],
                         result["emotion"], result["slots"])

    return {
        **state,
        "memory_snapshot": memory,
        "intent": result["intent"],
        "stage": result["stage"],
        "emotion": result["emotion"],
        "confidence": result["confidence"],
        "escalate": result["escalate"],
        "route": route
    }

# ── Knowledge Agent（接管 Step 6 + 8）─────────────────────────
@tool
def knowledge_agent(state: CustomerServiceState) -> CustomerServiceState:
    """Knowledge Agent：槽位提取 + 意图路由 RAG"""
    # 槽位提取
    slots = extract_slots(state["query"], state["intent"])

    # RAG 检索（按意图选择集合）
    collection_map = {
        "product_inquiry": "products",
        "faq": "faq",
        "policy_inquiry": "policies"
    }
    collection = collection_map.get(state["intent"], "faq")
    rag_results = chromadb.query(
        collection_name=collection,
        query_texts=[state["query"]],
        n_results=3
    )
    rag_results = [r for r in rag_results["documents"][0]
        if r.get("score", 0) >= 0.55]

    return {**state, "slots": slots, "rag_results": rag_results}

# ── Action Agent（接管 Step 6 + 8.5）──────────────────────
@tool
def action_agent(state: CustomerServiceState) -> CustomerServiceState:
    """Action Agent：槽位提取 + 函数执行（支持 Agent Loop）"""
    slots = extract_slots(state["query"], state["intent"])

    # 加载 functions.json 中的函数定义
    functions = load_functions()

    # Agent Loop：Think → Act → Observe → Decide（最多 3 次）
    loop_state = {"slots": slots, "results": [], "iteration": 0}
    for _ in range(3):
        think_result = think_step(loop_state, state["query"])
        if think_result["action"] == "respond":
            break
        act_result = act_step(think_result["function"], think_result["args"])
        observe_result = observe_step(act_result)
        decide_result = decide_step(observe_result, state["query"])
        loop_state["results"].append(observe_result)
        loop_state["iteration"] += 1
        if decide_result["action"] in ["respond", "ask_user"]:
            break

    return {**state, "slots": loop_state["slots"],
            "function_results": loop_state["results"]}

# ── Generator Agent（接管 Step 9 + 10）────────────────────
@tool
def generator_agent(state: CustomerServiceState) -> CustomerServiceState:
    """Generator Agent：综合所有结果生成最终回复"""
    result = generator_chain.invoke({
        "query": state["query"],
        "intent": state["intent"],
        "slots": state.get("slots", {}),
        "rag_results": state.get("rag_results", []),
        "function_results": state.get("function_results", {}),
        "history_summary": state["memory_snapshot"].get("summary", ""),
        "sop": load_sop(state["intent"])
    })

    return {**state, "reply": result["reply"], "memory_updates": result["facts"]}

# ── Memory Agent（接管 Step 0 + 10）──────────────────────
@tool
def memory_agent(state: CustomerServiceState) -> CustomerServiceState:
    """Memory Agent：三层记忆管理（短期/工作/长期）+ ChromaDB 持久化"""
    # 1. 短期记忆：写入当前对话上下文
    append_short_term(state["session_id"], state["query"], state["reply"])

    # 2. 工作记忆：超过 8 轮则压缩
    if get_conversation_turns(state["session_id"]) > 8:
        compress_to_working(state["session_id"])

    # 3. 长期记忆：持久化关键事实
    if state.get("memory_updates"):
        upsert_long_term(state["user_id"], state["memory_updates"])

    return state

7.4 LangGraph 流程编排

from langgraph.graph import StateGraph, END, START
from langgraph.checkpoint.memory import MemorySaver

# 构建图
graph = StateGraph(CustomerServiceState)

# 添加节点
graph.add_node("router", router_agent)
graph.add_node("knowledge", knowledge_agent)
graph.add_node("action", action_agent)
graph.add_node("generator", generator_agent)
graph.add_node("memory", memory_agent)

# 添加边
graph.add_edge(START, "router")

# Router 条件边：按路由分发
def route_decision(state: CustomerServiceState) -> str:
    if state.get("escalate"):
        return "escalate"
    route = state.get("route", "HYBRID")
    return route

graph.add_conditional_edges(
    "router",
    route_decision,
    {
        "CHITCHAT": "generator",    # 闲聊直接生成
        "KNOWLEDGE": "knowledge",    # 知识查询
        "ACTION": "action",         # 函数执行
        "HYBRID": "knowledge",       # HYBRID: 知识 + 函数并行
        "ESCALATE": "generator"      # 转人工
    }
)

# HYBRID 路由：knowledge 和 action 并行执行
graph.add_edge("knowledge", "generator")
graph.add_edge("action", "generator")
graph.add_edge("generator", "memory")
graph.add_edge("memory", END)

# 编译图（带 Checkpoint 持久化）
checkpointer = MemorySaver()
app = graph.compile(checkpointer=checkpointer)

# 推理入口
def chat(query: str, session_id: str, user_id: str) -> dict:
    config = {"configurable": {"thread_id": session_id}}
    result = app.invoke({
        "query": query,
        "session_id": session_id,
        "user_id": user_id,
    }, config)

    return {
        "reply": result["reply"],
        "intent": result["intent"],
        "route": result["route"],
        "escalate": result.get("escalate", False)
    }

7.5 Agent Loop 实现（Step 8.5 增强版）

@dataclass
class LoopStep:
    thought: str       # Think: 推理过程
    action: str         # Act: 要执行的函数
    args: dict         # Act: 函数参数
    observation: str    # Observe: 执行结果
    decision: str       # Decide: respond / continue / ask_user

def think_step(loop_state: dict, query: str) -> dict:
    """Think：LLM 决定下一步动作"""
    prompt = f"""当前状态：
- 已执行结果：{loop_state['results']}
- 槽位：{loop_state['slots']}
- 用户消息：{query}

请决定下一步动作：
- action: 要调用的函数名（如 respond 表示完成）
- args: 函数参数（如槽位缺失则 ask_user）
- reasoning: 推理过程"""
    result = llm.invoke(prompt)
    return parse_action(result)

def act_step(function_name: str, args: dict) -> dict:
    """Act：执行函数"""
    if function_name == "respond":
        return {"observation": "DONE"}
    func = get_function(function_name)
    return {"observation": func.invoke(args)}

def observe_step(act_result: dict) -> dict:
    """Observe：解析执行结果，注入槽位"""
    return act_result

def decide_step(observation: dict, query: str) -> dict:
    """Decide：判断是否继续或结束"""
    if observation.get("observation") == "DONE":
        return {"action": "respond"}
    if "缺少" in str(observation) or "不确定" in str(observation):
        return {"action": "ask_user"}
    return {"action": "continue"}

7.6 FastAPI 服务接入

from fastapi import FastAPI, HTTPException
from pydantic import BaseModel

app = FastAPI(title="食堂外卖智能客服 API")

class ChatRequest(BaseModel):
    query: str
    session_id: str
    user_id: str

class ChatResponse(BaseModel):
    reply: str
    intent: str
    route: str
    escalate: bool
    latency_ms: float
    tokens: int

@app.post("/chat", response_model=ChatResponse)
async def chat_endpoint(req: ChatRequest):
    start = time.time()
    try:
        result = chat(req.query, req.session_id, req.user_id)
        return ChatResponse(
            reply=result["reply"],
            intent=result["intent"],
            route=result["route"],
            escalate=result.get("escalate", False),
            latency_ms=(time.time() - start) * 1000,
            tokens=result.get("tokens", 0)
        )
    except Exception as e:
        raise HTTPException(status_code=500, detail=str(e))

@app.post("/reset/{session_id}")
async def reset_session(session_id: str):
    checkpointer.delete({"configurable": {"thread_id": session_id}})
    return {"status": "ok"}

八、总结与建议

8.1 核心结论

  1. 当前浣熊系统的串行 Pipeline 架构清晰、成本可控，适合快速验证
  2. 演进方向为 Hybrid Coordinator + LangGraph，兼顾性能与可维护性
  3. 演进策略为渐进式：Phase 1 Router 合并 → Phase 2 动态路由 → Phase 3 Multi-Agent
  4. 评测先行：用 Benchmark 量化改进效果，避免主观决策
  5. 框架选型：LangGraph 是当前最成熟的开源 Multi-Agent 编排框架，与现有 ChromaDB 无缝集成

8.2 优先级建议

优先级 行动项 周期 预期收益
P0 Phase 1 Router 合并（Step 0-5 → 一次 LLM） 2-4 周 P50 -30%，Token -20%
